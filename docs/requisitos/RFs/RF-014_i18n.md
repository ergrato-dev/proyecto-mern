## 📚 Conceptos Clave (para aprendices)

| Concepto          | Definición                                                                                |
| ----------------- | ----------------------------------------------------------------------------------------- |
| **i18n**          | Internacionalización — diseño del sistema para soportar múltiples idiomas sin refactoring |
| **l10n**          | Localización — proceso de añadir traducciones y formatos regionales a cada idioma         |
| **locale**        | Identificador de idioma/región. Ejemplos: `es`, `en`, `es-CO`, `en-US`                    |
| **namespace**     | Agrupación lógica de claves de traducción (usamos un único namespace `translation`)       |
| **interpolación** | Insertar variables dinámicas dentro de un texto traducido: `"Hola, {{name}}"`             |
| **fallback**      | Idioma al que se recurre si no se encuentra una clave en el idioma activo                 |

---

## Descripción General

El sistema NN Auth debe soportar dos idiomas: **español (`es`)** e **inglés (`en`)**.
Español es el idioma por defecto (proyecto SENA Colombia).
Los textos de la interfaz deben cambiar de forma dinámica sin recargar la página.
La preferencia se guarda en `localStorage` (cliente) y en la columna `locale` de la BD (servidor).

## Requisitos Funcionales

### RF-014.1 — Idiomas Soportados

El sistema debe soportar exactamente dos locales:

| Locale | Idioma  | Región           |
| ------ | ------- | ---------------- |
| `es`   | Español | Colombia (LATAM) |
| `en`   | English | General          |

**Archivo de traducción:** `src/locales/{locale}/translation.json`.

### RF-014.2 — Detección Automática del Idioma

Al cargar por primera vez (sin preferencia guardada), el frontend debe:

1. Leer `navigator.language` del navegador (ej: `"es-CO"`, `"en-US"`).
2. Extraer la parte base del locale (`"es"`, `"en"`).
3. Si el base locale está en la lista de soportados → usarlo.
4. Si no → usar `"es"` como fallback.

Herramienta: `i18next-browser-languagedetector`.

### RF-014.3 — Almacenamiento en localStorage

Toda preferencia de idioma (detectada o elegida manualmente) debe guardarse en
`localStorage` bajo la clave `i18nextLng`.

Al recargar la página, `i18next-browser-languagedetector` lee esa clave automáticamente.

### RF-014.4 — Columna `locale` en tabla `users`

Agregar columna en la tabla `users`:

```sql
ALTER TABLE users ADD COLUMN locale VARCHAR(10) NOT NULL DEFAULT 'es';
```

| Propiedad | Valor                                               |
| --------- | --------------------------------------------------- |
| Nombre    | `locale`                                            |
| Tipo      | `VARCHAR(10)`                                       |
| Default   | `'es'`                                              |
| Nullable  | `NOT NULL`                                          |
| Valores   | Solo `'es'` o `'en'` (validado en capa de servicio) |

### RF-014.5 — Endpoint PATCH /api/v1/users/me/locale

| Campo    | Valor                                       |
| -------- | ------------------------------------------- |
| Método   | `PATCH`                                     |
| Ruta     | `/api/v1/users/me/locale`                   |
| Auth     | Requerida (`Authorization: Bearer <token>`) |
| Body     | `{"locale": "es"}` o `{"locale": "en"}`     |
| Response | `UserResponse` actualizado (200 OK)         |
| Errores  | 401 (no autenticado), 422 (locale inválido) |

**Schema de request:**

```python
class UpdateLocaleRequest(BaseModel):
    locale: str

    @field_validator("locale")
    @classmethod
    def validate_locale(cls, v: str) -> str:
        if v not in ("es", "en"):
            raise ValueError("Locale no soportado. Use 'es' o 'en'.")
        return v
```

### RF-014.6 — Sincronización Frontend → Backend

**Al cambiar idioma (usuario autenticado):**

```
Usuario hace clic en LanguageSwitcher
  → i18n.changeLanguage(locale)  ← cambia la UI inmediatamente
  → localStorage.setItem("i18nextLng", locale)  ← persiste en cliente
  → PATCH /api/v1/users/me/locale { locale }  ← sincroniza en BD
```

**Al iniciar sesión:**

```
Login exitoso → Response incluye user.locale
  → i18n.changeLanguage(user.locale)  ← aplica preferencia guardada
  → localStorage.setItem("i18nextLng", user.locale)
```

### RF-014.7 — Componente LanguageSwitcher

El componente `LanguageSwitcher` debe:

- Mostrarse en la `Navbar` junto al `ThemeToggle`.
- Mostrar las dos opciones de idioma: "Español" / "English".
- Indicar visualmente el idioma activo (ej: texto resaltado o badge).
- Al seleccionar, llamar a `i18n.changeLanguage()` y sincronizar con el backend si hay sesión.
- Ser accesible: `aria-label="Selector de idioma"` y opciones con `lang` attribute.

### RF-014.8 — Catálogo de Traducciones

Todos los textos visibles de la interfaz deben estar en los archivos de traducción.
**Ningún texto debe estar hardcodeado** en los componentes.

**Estructura jerárquica de claves:**

```
nav.*                        → barra de navegación
common.*                     → textos reutilizables (loading, email, password, etc.)
auth.login.*                 → página de inicio de sesión
auth.register.*              → página de registro
auth.changePassword.*        → página de cambio de contraseña
auth.forgotPassword.*        → página de recuperación de contraseña
auth.resetPassword.*         → página de restablecimiento de contraseña
dashboard.*                  → panel de control
language.*                   → textos del selector de idioma
landing.*                    → landing page pública (hero, features, steps, stack, CTA, footer)
contact.*                    → formulario de contacto público
legal.backToHome             → enlace "Volver al inicio" compartido en LegalLayout
legal.footerCredit           → pie de página compartido en LegalLayout
legal.terms.*                → página Términos de Uso (9 secciones)
legal.privacy.*              → página Política de Privacidad (11 secciones)
legal.cookies.*              → página Política de Cookies (8 secciones + tabla de cookies)
```

> **Convención de secciones:** dentro de cada módulo legal, las claves siguen el patrón
> `legal.<módulo>.sN.heading` y `legal.<módulo>.sN.<elemento>` donde `N` es el número de sección.
> Las tablas (ej: cookies) usan claves dinámicas: `legal.cookies.s3.cookies.<nombre>.purpose`.

### RF-014.9 — Patrones de Uso en Componentes

#### Patrón 1 — Texto simple (`t()`)

Para cualquier cadena de texto sin markup HTML embebido:

```tsx
import { useTranslation } from "react-i18next";

function LoginPage() {
  const { t } = useTranslation();

  return (
    <div>
      <h1>{t("auth.login.title")}</h1>
      <p>{t("auth.login.subtitle")}</p>
      <button>{t("auth.login.submit")}</button>
    </div>
  );
}
```

#### Patrón 2 — Interpolación de variables (`{{variable}}`)

Para textos que contienen valores dinámicos:

```tsx
// JSON: { "dashboard": { "welcome": "Bienvenido, {{name}}" } }
const { user } = useAuth();
<h1>{t("dashboard.welcome", { name: user?.first_name })}</h1>;
// Renderiza: "Bienvenido, Carlos"
```

#### Patrón 3 — Texto con markup HTML (`<Trans>`)

Para textos que contienen elementos HTML embebidos (`<strong>`, `<a>`, `<code>`, etc.).
**No usar `dangerouslySetInnerHTML` — siempre usar `<Trans>` de react-i18next.**

```tsx
import { Trans, useTranslation } from "react-i18next";

// 1. Definir los elementos JSX que representan cada tag
//    Los nombres deben coincidir con los tags usados en el JSON
const S = <strong className="font-semibold text-gray-700 dark:text-gray-300" />;
const privacyLink = (
  <a href="/privacidad" className="underline text-blue-600" />
);

// 2. En el JSON, usar los nombres de tag como placeholders
// es/translation.json:
//   "legal.terms.intro": "La empresa <strong>NN</strong> ofrece el servicio.
//     Consulta la <privacyLink>Política de Privacidad</privacyLink>."

// 3. Renderizar con Trans, pasando los elementos como `components`
<Trans i18nKey="legal.terms.intro" components={{ strong: S, privacyLink }} />;
```

> **Regla:** el nombre de la clave en `components` (ej: `strong`, `privacyLink`) debe coincidir
> **exactamente** con el nombre del tag en el JSON. `<Strong>` ≠ `<strong>`.

#### Patrón 4 — Claves dinámicas (tablas o listas)

Para renderizar tablas o listas cuyo contenido varía por item:

```tsx
// JSON:
// "legal.cookies.s3.cookies": {
//   "access_token": { "purpose": "Autenticar al usuario", "duration": "15 min" },
//   "refresh_token": { "purpose": "Renovar la sesión",   "duration": "7 días" }
// }

const COOKIES = [{ name: "access_token" }, { name: "refresh_token" }] as const;

{
  COOKIES.map((cookie) => (
    <tr key={cookie.name}>
      <td>{cookie.name}</td>
      <td>{t(`legal.cookies.s3.cookies.${cookie.name}.purpose`)}</td>
      <td>{t(`legal.cookies.s3.cookies.${cookie.name}.duration`)}</td>
    </tr>
  ));
}
```

> **Importante:** al agregar un nuevo item al array, se debe agregar su clave correspondiente
> en **ambos** archivos JSON (`es/translation.json` y `en/translation.json`).

## Reglas de Negocio

| ID     | Regla                                                                                                                     |
| ------ | ------------------------------------------------------------------------------------------------------------------------- |
| RN-073 | El idioma por defecto del sistema es `es` (español).                                                                      |
| RN-074 | El sistema soporta exactamente dos locales: `es` y `en`.                                                                  |
| RN-075 | La preferencia de idioma se guarda siempre en `localStorage` (clave `i18nextLng`).                                        |
| RN-076 | Si el usuario está autenticado, el cambio de idioma se sincroniza con el backend via `PATCH /api/v1/users/me/locale`.     |
| RN-077 | Al restaurar sesión (token válido), se aplica el `locale` del usuario desde la BD.                                        |
| RN-078 | Los textos del `LanguageSwitcher` se muestran en el idioma propio de cada opción, no en el idioma activo.                 |
| RN-079 | El cambio de idioma es inmediato — sin recarga de página.                                                                 |
| RN-080 | El backend rechaza valores de `locale` distintos a `es` y `en` con HTTP 422.                                              |
| RN-081 | Nuevos usuarios se registran con `locale='es'` por defecto.                                                               |
| RN-082 | El `LanguageSwitcher` debe ser accesible: usa `aria-label` y atributo `lang` en opciones.                                 |
| RN-083 | Las claves de traducción se escriben en inglés, en `camelCase`.                                                           |
| RN-084 | Los valores de traducción se escriben en el idioma correspondiente al archivo.                                            |
| RN-085 | Las variables dinámicas en traducciones usan la sintaxis `{{variable}}` de i18next.                                       |
| RN-086 | Se usa un único namespace `translation` para todos los archivos (simplicidad pedagógica).                                 |
| RN-087 | El archivo de configuración de i18n (`src/i18n.ts`) importa los recursos de forma estática (no lazy-load).                |
| RN-088 | En tests, se usa un mock sincrono de i18next que retorna las traducciones en español.                                     |
| RN-089 | No se usa el locale con región (`es-CO`, `en-US`) — solo el base (`es`, `en`).                                            |
| RN-090 | Si falla la llamada PATCH al backend (error de red o token expirado), el cambio de idioma en el frontend igual se aplica. |

## Checklist — Cómo agregar i18n a una nueva página

Sigue estos pasos **en orden** cada vez que crees una nueva página o componente con texto visible:

### Paso 1 — Agregar las claves al JSON de español

Abre `fe/src/locales/es/translation.json` y agrega una nueva sección con todas las cadenas
de la página. Usa el nombre de la página como prefijo de clave.

```json
// Ejemplo para una nueva página "ProfilePage"
"profile": {
  "title": "Mi Perfil",
  "fullName": "Nombre completo",
  "email": "Correo electrónico",
  "saveButton": "Guardar cambios",
  "successMessage": "Perfil actualizado correctamente"
}
```

### Paso 2 — Agregar las mismas claves al JSON de inglés

Abre `fe/src/locales/en/translation.json` y agrega **las mismas claves** con los valores
en inglés. La estructura del JSON debe ser idéntica — solo cambian los valores.

```json
"profile": {
  "title": "My Profile",
  "fullName": "Full name",
  "email": "Email address",
  "saveButton": "Save changes",
  "successMessage": "Profile updated successfully"
}
```

> **Regla:** ambos archivos deben estar siempre sincronizados. Una clave en `es` sin
> su equivalente en `en` hará que aparezca la clave cruda (ej: `"profile.title"`) cuando
> el usuario cambie a inglés.

### Paso 3 — Declarar el hook en el componente

```tsx
import { useTranslation } from "react-i18next";

function ProfilePage() {
  const { t } = useTranslation();
  // ...
}
```

### Paso 4 — Reemplazar cada texto hardcodeado por `t("clave")`

```tsx
// ❌ Antes — texto hardcodeado
<h1>Mi Perfil</h1>
<button>Guardar cambios</button>

// ✅ Después — usando t()
<h1>{t("profile.title")}</h1>
<button>{t("profile.saveButton")}</button>
```

### Paso 5 — Para texto con markup HTML, usar `<Trans>`

Si una cadena contiene etiquetas como `<strong>`, `<a>`, `<code>` u otra:

```tsx
import { Trans, useTranslation } from "react-i18next";

// 1. Definir elemento JSX con las clases de estilo
const Bold = (
  <strong className="font-semibold text-gray-900 dark:text-gray-100" />
);

// 2. En el JSON: "profile.disclaimer": "Datos protegidos por la <strong>Ley 1581</strong>."

// 3. Renderizar con Trans
<Trans i18nKey="profile.disclaimer" components={{ strong: Bold }} />;
```

### Paso 6 — Actualizar el catálogo en este documento

Agrega la nueva sección al catálogo de claves en RF-014.8 (`**Estructura jerárquica de claves**`)
para que otros desarrolladores sepan que existe.

```
profile.*   → página de perfil de usuario
```

### Paso 7 — Verificar en el navegador

1. Con el idioma en **Español**: confirmar que los textos se ven correctamente.
2. Cambiar a **English**: confirmar que todos los textos cambian.
3. Si algún texto aparece como clave cruda (ej: `"profile.title"`), hay una clave faltante
   en el JSON — volver al Paso 1.
